# TalentLens — Логика AI анализа

> Документ описывает как работает движок анализа (`app/services/analysis.py`), что именно он считает, откуда берёт данные и как формирует оценки. Актуально для всех пяти режимов (A/B/C/D/E).

---

## Общая архитектура

```
AnalysisEngine
    ├── LLMProvider (Groq / OpenAI / Mock)
    ├── RetrievalService  ← BM25 + ChromaDB (исторические данные из всех проектов)
    ├── TeamContextService  ← живой состав команды из БД
    └── context_cache  ← 5-минутный кэш команды/отчётов; 1-часовой кэш результатов A/B/C
```

Каждый режим делает **один LLM-вызов**. Режимы A+B+C объединены в один вызов `position_intelligence` (экономия ~67% токенов).

---

## Источники данных

### 1. JD / Resume (ExtractedData)
Основной вход. Загружается из `extracted_data.structured_data` для нужного документа.

### 2. Исторические данные (Retrieval — BM25 + Vector)
Движок делает семантический запрос по всем документам проекта:
- **Типы**: `jd`, `job_request`, `report`, `interview`, `resume`
- **Лимит**: 3 000 токенов контекста
- Запрос строится автоматически под каждый режим:
  - Режим A/B/C: `"hiring requirements skills experience outcomes seniority history for: {JD text}"`
  - Режим D: `"candidate evaluation {name} skills match performance fit for: {JD text}"`
  - Режим E: `"actual work performed team responsibilities deliverables for: {JD text}"`

Таким образом, **в контекст LLM попадают данные из всех прошлых документов проекта** — резюме команды, недельные отчёты, интервью, предыдущие JD.

### 3. Команда (TeamContextService)
Актуальный состав команды из БД (`TeamMember` + связанные отчёты):
- Имена, роли, уровни, навыки
- Последние работы (из `ReportMemberLink` или `Document.team_member_id`)
- Командные наблюдения: блокеры, placed кандидаты

Контекст команды кэшируется **5 минут** (`context_cache`).

### 4. Интервью и фидбек клиента (Режим D)
Если у кандидата есть заметки из интервью или фидбек клиента, они добавляются в промпт с пометкой `"weight heavily for technical assessment"`.

## Режим A — Talent Brief

**Вход**: JD
**Что делает**: составляет бриф поиска — какие навыки нужны, с каким приоритетом, где искать.

### Как определяет criticality навыков
- Навыки, которые **уже есть в команде** → `"nice"` (не первостепенные)
- Навыки, которых **нет в команде** → `"must"` (реальный пробел)

Это ключевое: система не просто читает JD, а сравнивает требования с текущим составом команды и отмечает только настоящие пробелы как обязательные.

### Что возвращает
```json
{
  "skills_required": [{"name": "Kafka", "criticality": "must", "market_availability": "hard", "notes": "..."}],
  "search_guidance": ["..."],
  "historical_insights": ["..."],
  "pitfalls": ["..."],
  "estimated_time_to_fill_days": 30
}
```

## Режим B — Historical Match

**Вход**: JD
**Что делает**: ищет похожие позиции из прошлого и извлекает паттерны успеха/неудачи.

Данные берутся **исключительно из retrieval-контекста** — то есть из документов, которые уже загружены в проект. Если прошлых данных нет, массивы `similar_positions`, `success_patterns`, `failure_patterns` будут пустыми.

Каждый инсайт должен содержать ссылку на источник `[Source N]` — правило анти-галлюцинации.

---

## Режим C — Level Advisor

**Вход**: JD
**Что делает**: рекомендует уровень сеньорности (junior/mid/senior/lead).

Учитывает:
- Явные требования JD (лет опыта, ответственности)
- Текущий состав команды — рекомендует уровень, который **балансирует композицию** (например, если уже есть lead, не рекомендует ещё одного)
- Исторические кейсы из retrieval-контекста

---

## Режим D — Candidate Scorer (главный режим)

**Вход**: резюме кандидата + JD
**Что делает**: детерминированная 7-шаговая оценка кандидата от 0 до 100.

### Детерминированная формула (DEC-026)

```
raw_score = (hard_skills × 0.40)
          + (experience × 0.25)
          + (domain × 0.15)
          + (soft_skills × 0.10)
          + (team_fit × 0.10)
```

#### ШАГ 1 — Role Alignment Gate (Ворота совпадения роли)

Первым делом система определяет **тип роли кандидата** (по тому, что он **делал**, а не чем управлял):
- `engineer/developer` — пишет код, строит системы (IC)
- `manager` — управляет людьми, проектами, сроками
- `analyst` — анализирует, пишет отчёты, но не строит пайплайны
- `designer`, `qa`, `devops`, `data_scientist`, `other`

И **тип роли JD** — из раздела responsibilities.

**Если типы не совпадают** (например, кандидат — менеджер, а JD нужен инженер):
- `role_alignment_score = 10`
- `overall_score = min(raw_score, 35)` — жёсткий кап
- `verdict = "not_recommended"` (принудительно)

**Если совпадают**:
- `role_alignment_score = 100`
- Никакого капа нет

**Фейлсейф**: если LLM вернул неполный ответ и `role_alignment` не заполнен, дефолтное значение `role_alignment_score = 10` — система автоматически считает это несовпадением роли.

---

#### ШАГ 2 — Hard Skills (вес 40%)

Система перебирает каждый **must-have навык из JD** и проверяет кандидата:

| match_level | Коэффициент | Критерий |
|-------------|-------------|----------|
| `hands_on` | 1.0 | Кандидат **использовал** навык как IC: строил, писал, отлаживал |
| `exposure` | 0.2 | Навык упомянут, но только в контексте управления ("managed a team using Python") |
| `none` | 0.0 | Не упоминается совсем |

```
hard_skills_score = (сумма коэффициентов / количество must-have навыков) × 100
```

**Важное правило**: "Managed a team using Python" ≠ "Knows Python". Это `exposure (0.2)`, а не `hands_on (1.0)`.

---

#### ШАГ 3 — Experience (вес 25%)

Считаются только **годы в той же категории роли**, что требует JD:
- JD нужен инженер → считаются только годы как инженер/IC
- Годы в роли PM = 0 инженерных лет
- Годы как аналитик = 0 инженерных лет

Сравнение с требованием JD:

| Выполнение требования | Балл |
|-----------------------|------|
| 100%+ (соответствует или превышает) | 100 |
| 75–99% | 75 |
| 50–74% | 50 |
| 25–49% | 25 |
| < 25% или полностью не та роль | 10 |

**PM-опыт не засчитывается как инженерный, даже если PM работал в технических проектах.**

---

#### ШАГ 4 — Domain & Context (вес 15%)

| Критерий | Баллы |
|----------|-------|
| Та же индустрия, что и клиент JD | +40 |
| Похожий размер/стадия компании | +20 |
| Релевантные domain-знания | +40 |
| Максимум | 100 |

---

#### ШАГ 5 — Soft Skills (вес 10%)

| Критерий | Максимум |
|----------|----------|
| Communication — свидетельства коммуникации | 35 |
| Collaboration — командная работа | 35 |
| Problem-solving — примеры решения проблем | 30 |
| Итого | 100 |

---

#### ШАГ 6 — Team Complementarity (вес 10%)

| Критерий | Баллы |
|----------|-------|
| Заполняет реальный пробел в команде | +50 |
| Приносит экспертизу, которой нет в команде | +30 |
| Уровень сеньорности подходит структуре команды | +20 |
| Максимум | 100 |

Если данных о команде нет — дефолт 50.

---

#### ШАГ 7 — Итоговый расчёт

```
raw_score = (hard × 0.40) + (exp × 0.25) + (domain × 0.15) + (soft × 0.10) + (team × 0.10)

if role_alignment_score < 50:
    overall_score = min(raw_score, 35)
else:
    overall_score = round(raw_score)
```

### Вердикты

| Балл | Вердикт |
|------|---------|
| 85–100 | `strong_fit` |
| 65–84 | `moderate_fit` |
| 45–64 | `risky` |
| < 45 | `not_recommended` |
| ≤ 35 (role mismatch) | `not_recommended` (принудительно) |

### Pydantic-валидатор (Post-LLM коррекция)

После получения ответа от LLM срабатывает `@model_validator(mode="after")`:
1. Если `role_alignment_score < 50` → кап 35, форс `not_recommended`
2. Иначе → проверяет соответствие вердикта баллу и исправляет если LLM ошибся

## Режим E — JD Reality Check

**Вход**: JD
**Что делает**: проверяет JD на соответствие реальности — что уже есть в команде, что действительно нужно, оправдан ли найм.

### Три блока анализа

**1. Skills vs Reality**
- `jd_requires` — что написано в JD
- `team_already_has` — что уже покрывает команда (из TeamContextService)
- `actually_needed` — что реально нужно нанять
- `questionable_requirements` — требования JD, которые не соответствуют реальному стеку

**2. Workload Analysis**
- Что утверждает JD о нагрузке
- Что показывают недельные отчёты команды (из retrieval)
- Несоответствия

**3. Necessity Check**
- `is_hire_justified: bool`
- Обоснование на основе отчётов
- Альтернативные предложения
- Приоритет: `critical / high / medium / low`

---

## Антигаллюцинационные правила (применяются во всех режимах)

Injected в каждый промпт:

1. Анализ только на основе предоставленных документов
2. Цитировать `[Source N]` для каждого утверждения
3. Если данных недостаточно — сказать явно, не угадывать
4. Рейтинг уверенности: `LOW / MEDIUM / HIGH`
5. Ответ только в JSON

## Кэширование

| Кэш | TTL | Что хранит |
|-----|-----|-----------|
| `context_cache` (team) | 5 мин | строка TeamContextService |
| `context_cache` (reports) | 5 мин | строка RetrievalService для отчётов |
| AnalysisResult в БД (A/B/C) | 1 час | готовые результаты режимов, привязанные к JD документу |

Режимы A/B/C: если за последний час уже был расчёт по этому JD — возвращается кэш из БД без вызова LLM.

## Как работает `relevant_years` на практике

Пример из реальных данных проекта:

| Кандидат | resume.years_of_experience | Что реально считается | experience_score |
|----------|---------------------------|----------------------|-----------------|
| Yuki Tanaka | 9 лет | 9 лет инженер (Adyen + Rakuten) | 100 |
| David Kim | 5 лет | 5 лет инженер (e-commerce) | 75–100 (если JD требует 5+) |
| Marcin Zając | 2 года | 2 года, vs 5+ требуется (40%) | 25 |

Если кандидат — менеджер с 10 годами опыта, но JD нужен инженер:
- `relevant_years = 0` (инженерных лет нет)
- `experience_score = 10`
- `role_alignment_score = 10` → `overall_score ≤ 35`
